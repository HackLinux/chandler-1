How to read Chandler's automatic documentation

Background

Chandler's automatic documentation gives details about Chandler's Data Model and Content Model. The Data Model provides a base set of Kinds, Attributes, and other types which are drawn on to create new Kinds and Attributes using a consistent vocabulary. The Content Model builds on these base types to create types for different areas of Chandler functionality.

The Data Model and the Content Model are defined in a collection of XML files called parcels. These XML files are automatically transformed into documentation. See the repository intro for more about the format of the parcel.xml files.

Parcel Index

The index page for the latest Content Model documentation is available at http://www.osafoundation.org/docs/current/model/, these pages are updated every three hours.

Each row of the index refers to a particular parcel, with links to various views of that parcel. Each parcel's summary page for is meant to give an overview of the Kinds defined in that parcel. There are also pages that describe the parcel's Kinds, Attributes, Enumeration, Aliases, and Types in detail. If a row doesn't have a link to one of these base types, it means that parcel didn't define anything of that type.

Summary Page

Each parcel's summary page lists gives the parcel's general description, then summarizes the Kinds the parcel defines. The beginning of each Kind summary will say something like: "Content Item is a Kind defined in Chandler Content Model" followed by any description, examples, or issues associated with the Kind.

Next, all the attributes the parcel specifies for the Kind (but not its inherited attributes) are listed, with a sentence explaining the attribute's type, cardinality, and whether it's a reference to some other Kind or normal value. For instance, "Projects is a list of references to Project items." Next, the attribute's description or any issues associated with it are listed.

The summary page is meant to give a quick overview of the parcel, but it doesn't say everything, so the Kinds and Attributes listed on the summary page link to the more detailed Kinds and Attributes pages.

Kinds Page

The Aliases, Enumerations, and Types pages are fairly self explanatory, but the Kinds page has enough going on that it deserves a little attention.

All Kinds have superKinds, the default superKind is Item. At the top of each parcel's Kinds page is a tree describing its Kinds' inheritance structure, with a link to each Kind lower down on the page.

The detail section for each Kind gives the Kind's general description, then a table with a row for all the attributes associated with the Kind, including any attributes inherited from its superKind. The columns of the table list common aspects:

  • displayName - the full name of the Attribute
  • itemName - the name used to refer to the attribute, which in fact is not an aspect
  • type - the Kind of the Attribute
  • cardinality - the Attribute's cardinality
  • inverseAttribute - if the Attribute is a reference, the Attribute it refers to
  • superAttribute - the Attribute's parent, if it has one
  • required - whether the attribute is required. By default, Attributes are not required
  • defaultValue - some attributes define a default value the Attribute should be set to

Known Bugs

Documentation is generated by running XSL transforms against parcel.xml files. Known bugs in this process:

  • The XSL may die a painful death if a Kind has more than one superKind.
  • At one point some algorithms assumed that itemNames were globally unique. Most code no longer assumes this, but vestiges of the old may remain, this hasn't been tested thorougly enough yet.
  • The XSL transforms use EXSL functions. libxslt's EXSL implementation is known to work, other implementations haven't been tested.

If you believe you've found a bug, please make a bug report on the OSAF wiki.
